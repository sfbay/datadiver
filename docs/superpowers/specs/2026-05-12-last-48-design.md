# The Last 48 — Design Spec

**Date:** 2026-05-12
**Status:** Draft
**Route:** `/live-feeds` (replaces existing scanner-only view)
**Accent:** `#d4a435` (ochre-500 — retains the warm yellow "live" signal already in the nav)
**Supersedes:** `2026-03-17-live-scanner-feeds-design.md` (the scanner-radio LiveFeeds view, which becomes a sub-feature of The Last 48)

---

## Overview

The Last 48 is the evolution of DataDiver's Live tab. Instead of a single-purpose scanner-radio view, the route becomes a **comprehensive map-based surface for the freshest civic data flowing into SF's public datasets** — interactive for journalists scanning recent activity, and headless-capable as a kiosk display that conveys a sense of life and vitality without a human at the keyboard.

The brand name is editorial-strategic: "The Last 48" tells the truth out loud — *we are looking backward, not forward* — because SF's open data is published with intrinsic processing lag. The freshest dataset (`gnap-fj3t`, "911 Realtime") has events ~7 hours old; most others are 12–24 hours behind. Calling the view "Live" overpromises; calling it "The Last 48" makes the lag part of the editorial frame rather than a footnote.

### Design Philosophy

- **Honest latency.** Every chrome element shows real-time freshness data: when each dataset was last published AND how old the freshest event is. The reader always knows what "now" actually means per source.
- **Two lenses on one window.** All datasets share a 48-hour sliding window. The user picks how to *look* at it: FLOW (animated event arrival) or HOTSPOTS (static anomaly choropleth).
- **Director-driven kiosk.** A meta-mode rotates through FLOW + HOTSPOTS + headline scenes on a 5-minute loop, giving the surface autonomous life for unattended displays.
- **Scanner audio as soundtrack.** The existing scanner-radio feeds remain available as a compact launcher tile, NOT an embedded player — leaning into the user's pattern of "scanner running in one tab, visual data in another."

### Audience

Same as DataDiver overall — journalists, civic researchers, advocates. New audience implied by kiosk mode: ambient civic-data displays in newsrooms, community spaces, university campuses, and the user's own 55-inch portrait monitor (per memory: `user_background.md`).

## Data Sources & Latency Tiers

Measured against the live Socrata API on 2026-05-12 15:07 UTC. Latency varies dramatically by dataset; the architecture must respect this rather than paper over it.

### Tier 1 — Live-eligible (4–18h event lag)

| Dataset | ID | Date field | Refresh cadence | Event lag | Poll every |
|---|---|---|---|---|---|
| 911 Realtime | `gnap-fj3t` | `received_datetime` | Continuous (1-min refreshes) | ~7h | 2 min |
| Fire/EMS Dispatch | `nuek-vuh3` | `received_dttm` | Every few hours | ~12h | 30 min |
| 311 Cases | `vw6y-z8j6` | `requested_datetime` | Every few hours | ~15h | 30 min |
| 911 Historical | `2zdj-bwza` | `received_datetime` | Every few hours | ~15h | 30 min |
| Parking Revenue | `imvp-dq3v` | `session_start_dt` | Daily-ish | ~17h | 4h |

### Tier 2 — Marginal (1–3 day event lag)

| Dataset | ID | Date field | Refresh cadence | Event lag | Poll every |
|---|---|---|---|---|---|
| Police Incidents | `wg3w-h783` | `incident_datetime` | Daily (business days?) | ~39h | 6h |

Tier 2 datasets are included in the 48h window but visually de-emphasized — their freshest events are typically older than the 48h cutoff but still contribute to HOTSPOTS baselines.

### Tier 3 — Excluded (weeks-to-months event lag)

| Dataset | Event lag | Disposition |
|---|---|---|
| Traffic Crashes (`ubvf-ztfx`) | ~73 days | **Not in The Last 48.** Remains on `/traffic-safety` only. |
| Fire Incidents (`wr8u-xric`) | ~83 days | **Not in The Last 48.** Remains on `/emergency-response` only. |
| Parking Citations (`ab4h-6ztd`) | data-quality bug (future dates) | **Not in The Last 48.** Geo gap already documented. |

### Refresh cadence — methodological note

No dataset advertises an explicit `update frequency` field in its Socrata metadata. The polling intervals above are *inferred* from observed `rowsUpdatedAt` patterns measured during design. The hook must tolerate cadence drift (sometimes a "30-min" dataset goes 90 min between refreshes). A backoff strategy on cache miss prevents excessive polling.

## Information Architecture

### Route

```
/live-feeds                                         → FLOW (default mode)
/live-feeds?mode=hotspots                           → HOTSPOTS
/live-feeds?mode=kiosk                              → KIOSK director (landscape 16:9 default)
/live-feeds?mode=kiosk&orientation=portrait         → KIOSK director (portrait 9:16)
```

Existing app-wide URL state (date range, dark mode, neighborhood selection) continues to apply where relevant. Date range is *overridden* on this view — Last 48 always queries against its own 48h window, regardless of the global date range. This is the only DataDiver view that ignores the global date picker.

### URL State Inventory

| Param | Values | Default | Notes |
|---|---|---|---|
| `mode` | `flow` \| `hotspots` \| `kiosk` | `flow` | URL-shareable mode selection |
| `orientation` | `landscape` \| `portrait` | `landscape` | Kiosk-only |
| `datasets` | comma-list of dataset slugs | all Tier-1 on | Filter chips at top |
| `n` | neighborhood slug | none | Existing pattern, optional |
| `kiosk-pause` | `1` (string) | absent | Stops director rotation in kiosk |

### Migration: existing scanner-radio LiveFeeds

The 570-line `src/views/LiveFeeds/LiveFeeds.tsx` (scanner radio view) is **not deleted** but **reduced and relocated**. Its current content (scanner feeds grouped by service, SFPD/SFFD districts, contextual neighborhood data) is decomposed:

- **Feed launcher logic** → `src/views/Last48/chrome/ScannerStrip.tsx` (compact strip along bottom of map)
- **District/battalion context** → optional drill-in modal launched from the strip
- **The route itself (`/live-feeds`)** → replaced by `Last48.tsx`

The decision to retain `/live-feeds` as the URL (rather than introduce `/last-48`) preserves bookmark continuity and avoids a redirect.

- Map area: `flex-1`, minimum height 480px

When `mode=kiosk`, the layout collapses to:

- Header: hidden (replaced by scene-transition indicator strip)
- Filter chips: hidden (director picks datasets per scene)
- Right rail: hidden by default, can appear briefly as part of a scene
- Scanner strip: hidden (scanner audio is meant for a separate tab)
- Map: fills nearly the entire viewport

In portrait (`orientation=portrait`), the same content reflows: the right rail (if shown for a scene) moves to the bottom; the headline-stats area moves to the top.

## Data Engine

### Central hook: `useLast48Window`

The architectural heart of the view. One hook polls all Tier-1 (and optionally Tier-2) datasets at their individual cadences, maintains an in-memory sliding 48h buffer keyed by `(datasetId, eventId)`, and exposes a normalized event stream to consumers.

```ts
// src/hooks/useLast48Window.ts

export type Tier = 'tier1' | 'tier2'

export interface NormalizedEvent {
  id: string                  // datasetId + native row id
  datasetId: DatasetId
  timestamp: string           // ISO; the dataset's "received" or "issued" time
  receivedAt: number          // unix ms, for sorting
  neighborhood?: string       // SF Analysis Boundary if available
  longitude?: number
  latitude?: number
  callType?: string
  severity?: 'low' | 'medium' | 'high'  // dataset-specific bucketing
  raw: Record<string, unknown>          // original Socrata row, for detail panels
}

export interface Last48WindowResult {
  events: NormalizedEvent[]              // sorted newest-first
  byDataset: Record<DatasetId, NormalizedEvent[]>
  freshness: Record<DatasetId, {
    rowsUpdatedAt: number                // when Socrata last refreshed
    maxEventTime: number                 // freshest event in dataset
    eventLagMs: number                   // now - maxEventTime
    refreshLagMs: number                 // now - rowsUpdatedAt
  }>
  isLoading: boolean                     // true on first load only
  isPolling: boolean                     // true while ANY dataset is mid-fetch
  error: Record<DatasetId, string | null>
  refetch: () => void
}

export function useLast48Window(opts: {
  datasets: DatasetId[]                  // which to include (from filter chips)
  tier: Tier                             // 'tier1' default; 'tier2' adds Police Incidents
}): Last48WindowResult
```

### Polling architecture

Each dataset has its own polling interval (defined per the table above). Implementation:

```ts
// src/hooks/usePollCadence.ts

const POLL_INTERVALS: Record<DatasetId, number> = {
  '911-realtime': 2 * 60 * 1000,        // 2 min
  'fire-ems-dispatch': 30 * 60 * 1000,  // 30 min
  '311-cases': 30 * 60 * 1000,
  '911-historical': 30 * 60 * 1000,
  'parking-revenue': 4 * 60 * 60 * 1000, // 4h
  'police-incidents': 6 * 60 * 60 * 1000, // 6h (tier2)
}

// Returns a per-dataset poll handle that fires `queryFn` on its cadence,
// with jitter to avoid all datasets pinging Socrata at the same second.
// Pauses when `document.hidden` (tab not visible) — resumes on visibility change.
// Exponential backoff on consecutive failures, capped at 30 min.
```

### Sliding 48h buffer

The hook maintains an in-memory Map keyed on `(datasetId, eventId)`. On each poll:

1. Query `WHERE <dateField> >= now() - 48h` (server-side filter)
2. Deduplicate against existing buffer (events already seen are skipped)
3. Evict any events older than the 48h cutoff
4. Recompute aggregations (top-N anomalies for HOTSPOTS, sorted list for FLOW)

This is **not** a Zustand store — it lives in a `useRef` inside `useLast48Window` to avoid causing re-renders on every poll. Consumers subscribe via `useSyncExternalStore` to a stable snapshot.

### Tab-visibility behavior

When the tab is hidden (`document.visibilityState === 'hidden'`), polling pauses. On return, all datasets are re-fetched once before resuming their cadences. This avoids burning Socrata quota for closed tabs and ensures a fresh view on return.

### Anomaly baseline for HOTSPOTS

```ts
// src/hooks/useAnomalyBaseline.ts

// For each neighborhood × dataset, computes:
//   - typical event count per 48h window (baseline)
//   - standard deviation of that count
//   - current 48h count and resulting z-score
// Baseline window: trailing 12 weeks of 48h windows from the SAME dataset
//   (not overlapping; rolled forward at 48h granularity).
// Computed server-side via a single Socrata GROUP BY query per dataset,
// cached for the duration of the session (refreshed only when the
// hook is first instantiated).
```

The baseline window of **12 weeks** is editorially calibrated:

- Too short (4 weeks): noisy, single events can shift the mean
- Too long (52 weeks): obscures seasonal patterns relevant to civic data (e.g., 311 illegal-dumping spikes in summer)
- 12 weeks: 84 days = 42 non-overlapping 48h windows, enough N for a defensible σ

## Mode: FLOW

The default mode. Animated event arrival on the map; chronological event list in the right rail; scrubber accessible along the bottom for accelerating into timelapse motion.

### Visual treatment

- **Map dots:** each event from the 48h buffer appears as a small filled circle at its geo coordinates. Pigment = dataset color (consistent with existing per-dataset palette: brick for fire, indigo for 911, terracotta for 311, etc.).
- **Newest-event pulse:** when a new event is added to the buffer (via poll), it pulses 3× then fades to its resting state. The pulse is the visual reward of "fresh data arrived."
- **Trail decay:** events fade in opacity from 1.0 → 0.4 as they age across the 48h window. Oldest events are visible but de-emphasized.
- **Hover:** existing `useMapTooltip` pattern; tooltip shows time, dataset, neighborhood, snippet.
- **Click:** opens an inline detail panel (right side, slides over rail) — uses `DetailPanelShell` from the existing pattern.

### Right rail: chronological event list

- Newest at top, scrolls automatically as new events arrive
- Each row: time · dataset chip · neighborhood code · one-line snippet
- Click a row → flies the map to that event AND opens detail panel
- Max 50 visible rows; older events accessible by scrolling

### Scrubber + timelapse access

A subtle scrubber strip runs along the bottom of the map area:

```
SUN 14:00 ······························· NOW (●)    [▶▶] [↻]
```

- Default position: NOW (rightmost) — live edge
- Drag the indicator → time cursor moves; map shows only events within ±15 min of that cursor
- Click `▶▶` → "timelapse mode" — cursor snaps to start of 48h window, plays forward at 4× speed, then stops at NOW. User can change speed via repeated `▶▶` clicks (cycles 1× → 2× → 4× → 8×).
- Click `↻` → loop mode (timelapse restarts on completion)

This is the unification of what was originally proposed as separate STREAM and TIMELAPSE modes — same engine, two presets accessed through the same control.

### Mobile / narrow viewport

At <1024px width, the right rail collapses to a chevron (existing `MapSidebar` primitive). The scrubber moves above the scanner strip to avoid overlap. Filter chips wrap to two rows.

## Mode: HOTSPOTS

A static spatial snapshot answering "which neighborhoods stood out from normal during the last 48 hours?"

### Visual treatment

- **Map:** SF neighborhood polygons (from existing `useNeighborhoodBoundaries`), choropleth-filled by z-score of current-window event count vs baseline.
- **Threshold:** only neighborhoods with `|z| ≥ 0.5` get any color treatment at all. Below this threshold, the polygon stays transparent — no fill, no outline. Avoids "every neighborhood looks slightly elevated."
- **Color scale (above threshold):** divergent on the existing earth-tone palette.
  - `z ≥ +0.5` → escalating fills: ochre (+0.5 to +1.0) → terracotta (+1.0 to +2.0) → brick (>+2.0)
  - `z ≤ −0.5` → faint paper-tone fill with dashed outline (signals "unusually quiet" without competing visually with the high-anomaly side)
- **Hover:** tooltip with raw count, baseline mean, σ, z-score, percentile rank.
- **Click:** flies to the neighborhood AND opens the `NeighborhoodCensusContext` panel + a per-dataset breakdown ("311: +2.4σ · Fire/EMS: −0.3σ · 911: +0.8σ"), so the user can see *which* dataset is driving the anomaly.

### Right rail: ranked anomaly list

- Ordered by absolute z-score, most extreme first
- Each row: rank · neighborhood · z-score (color-coded — brick high-positive, paper neutral, moss high-negative)
- Click → same drill-in as map click
- Footer link: "Methodology — what σ means here" → opens a small popover explaining the 12-week baseline

### Dataset filter behavior in HOTSPOTS

The dataset filter chips at top determine which datasets contribute to the z-score. If only "911" is selected, the choropleth shows anomalies in 911 calls specifically. If multiple datasets are selected, z-scores combine via weighted sum across selected datasets (weight = inverse of baseline variance — i.e., datasets with more stable baselines contribute more).

### Why not just "raw count heatmap"

Decision documented in brainstorm 2026-05-12: a pure density heatmap is mostly useless because downtown is always busy and the heatmap looks the same every day. The anomaly framing makes the visualization tell a story — "Visitacion Valley's fire calls are 3× normal this 48 hours." That's a headline; raw density is not.

## Detail Panels — "Quick peek, deep dive"

The interaction pattern that makes The Last 48 a *scanning surface* rather than a *competing investigation tool*: clicking anything on the map opens a compact "peek" panel; the panel's footer links the user out to the dedicated dataset view for full investigation.

### Why this matters editorially

Every existing per-dataset view in DataDiver (`/emergency-response`, `/cases-311`, `/dispatch-911`, etc.) already implements investigation tooling — filters, histograms, trend baselines, neighborhood rankings. The Last 48 must not re-implement those; it must *gateway* to them. Detail panels are the seam between scanning and investigation.

### Layer 1 — Shared shell (reuse)

All detail panels reuse the existing `DetailPanelShell` chrome — slide-in-from-right behavior, close button, `z-30` stacking. No changes to the shell required.

### Layer 2 — Event peek (FLOW)

A new `<Last48EventPeek>` component adapts to whichever dataset the clicked event belongs to. Intentionally **compact** — not a full investigation, just enough to know what you're looking at.

Body layout:
- Eyebrow: dataset chip + time
- Headline: dataset-specific field (call type, service request, etc.)
- Location line: neighborhood · cross streets (if available)
- Static map preview (small): single location dot on a tile snippet
- 3–4 dataset-specific fields
- Footer link: `Explore <Dataset Name> →`

Dataset-specific headline + fields:

| Dataset | Headline | Compact-panel fields |
|---|---|---|
| 911 Realtime / Historical | Call type | Disposition · Unit · Response time · Priority |
| Fire/EMS Dispatch | Call type | Unit · Response time · Station · Battalion |
| 311 Cases | Service request type | Status · Source · Service area · Case ID |
| Parking Revenue | Meter location | Amount · Method · Duration · Meter ID |
| Police Incidents | Incident category | Subcategory · Resolution · Report type · Report ID |

### Layer 3 — Neighborhood peek (HOTSPOTS)

A new `<Last48NeighborhoodPeek>` component opens when the user clicks an anomaly-colored polygon in HOTSPOTS. Diagnoses the anomaly rather than just naming it.

Body layout:
- Eyebrow: `── NEIGHBORHOOD ANOMALY +Nσ above` (or `unusually quiet  −Nσ below`)
- Headline: neighborhood name + one-line caption ("unusual activity in the last 48 hours" / "below baseline activity in the last 48 hours")
- Per-dataset z-score breakdown — bar chart, bar width by absolute z, color by direction (terracotta positive, paper negative)
- Top contributing events — 3–5 specific events from the 48h buffer, sorted by their contribution to the anomaly. Each row clickable → opens the event peek (layer 2) in place.
- Demographic context — reuses existing `<NeighborhoodCensusContext>` component (currently used on the Demographics view)
- Footer link: `See <Neighborhood> across SF →` (links to neighborhood-filtered Home, or a dedicated neighborhood-detail route if that ships first)

### Deep-link contract with destination views

The footer "Explore" link takes the user to the dataset's dedicated view *with the specific event pre-selected* via query-param. Destination views must support a deep-link param if they don't already:

| Source event | Link target | Param |
|---|---|---|
| 311 case | `/cases-311?case=<id>` | `case` |
| 911 dispatch (either dataset) | `/dispatch-911?incident=<id>` | `incident` |
| Fire/EMS dispatch | `/emergency-response?incident=<id>` | `incident` |
| Parking Revenue session | `/parking-revenue?meter=<id>` | `meter` |
| Police incident | `/crime-incidents?incident=<id>` | `incident` |

Audit of existing deep-link support is a **precondition for Phase 1's FLOW interactivity**. Where a destination view doesn't support deep-linking yet, that's a small add (~30 lines per view) that lands as part of Last 48 work, not separately. Estimated 1 day total across all five datasets.

### Why "compact" rather than "rich" peek

A rich peek would duplicate functionality from the dedicated views and create maintenance debt — every change to a per-dataset panel would need to land twice. The compact peek's job is **recognition**, not **investigation**: "What is this thing? Where is it? Do I want to dig deeper?" If yes, one click sends them to the right tool.

## Mode: KIOSK (meta-mode)

KIOSK is conceptually different from FLOW and HOTSPOTS — it's an autonomous **director** that composes scenes from the other modes (and standalone "headline" scenes) into a rotating loop. The user does not interact with it; it runs unattended on a display.

### Scene script

The director cycles through scenes on a default ~5-minute loop:

| # | Scene | Duration | Content |
|---|---|---|---|
| 1 | **Headline card** | 20s | Big number: most-extreme z-score anomaly across all datasets. "311 calls in MISSION are +2.4σ above typical." |
| 2 | **FLOW · last 30 min** | 60s | FlowMode focused on most recent 30 min; map zoom = citywide; auto-pans to follow newly-arrived events |
| 3 | **HOTSPOTS** | 45s | Static choropleth; rail visible briefly to show the ranking |
| 4 | **FLOW · timelapse** | 90s | FlowMode at 4× speed, playing the full 48h window |
| 5 | **Scene focus: top anomaly neighborhood** | 60s | HOTSPOTS-style choropleth, but zoomed in on the #1-anomaly neighborhood; sidebar shows per-dataset breakdown |
| 6 | **Freshness honesty card** | 15s | "Data through HH:MM PT" stamp; shows event-lag per dataset as a small chart |
| — | Loop | | |

Total ≈ 290s (just under 5 min). Each scene has its own fade-in/out transition (~600ms cross-fade between scenes).

### Director implementation

```ts
// src/views/Last48/kiosk/KioskDirector.tsx

interface Scene {
  id: string
  durationMs: number
  render: (api: SceneAPI) => React.ReactNode
}

interface SceneAPI {
  events: NormalizedEvent[]       // from useLast48Window
  freshness: FreshnessMap
  anomalies: AnomalyResult[]      // from useAnomalyBaseline
  focusNeighborhood?: string      // for the focus scene
  pause: () => void
  resume: () => void
}

// Scene sequencer: setTimeout chain, advances on duration completion.
// Pause-on-?kiosk-pause=1 for debugging / screenshot capture.
```

### Aspect ratio handling

- `orientation=landscape` (default): 16:9 layout — map fills most of viewport, headline strip top, scene indicator bottom
- `orientation=portrait`: 9:16 layout — map fills upper 2/3, scene-specific content stack in lower 1/3

Layout uses `clamp()` and viewport-relative units to scale headline typography across screen sizes (per the liquid layout pattern).

### Auto-pause when no fresh data

If no new events have arrived in 30+ min (rare but possible during quiet hours), the director silently extends the FLOW timelapse scene rather than showing repetitive "nothing new" frames. This keeps the screen visually alive even during dead hours.

### Pause / debug

`?kiosk=1&kiosk-pause=1` stops scene rotation at the current scene — useful for screenshots, debugging, and ad-hoc presentations.

## Scanner Integration (launcher strip)

The existing scanner-radio data (`src/data/scannerFeeds.ts`, `SCANNER_FEEDS`, `FEED_SOURCES`) is retained in full. What changes is *presentation*: instead of a 570-line dedicated view, scanner content becomes a **compact launcher strip** at the bottom of The Last 48's map area.

### Strip behavior

- Strip is `h-12` always visible (except in kiosk mode)
- Shows: 📡 SCANNER · [currently-relevant feed name based on neighborhood selection or freshest-event location] · ▶ TUNE IN
- Click `▶ TUNE IN` → opens the underlying audio stream URL in a **new browser tab** (does NOT embed audio in The Last 48)
- The "currently relevant" feed is selected by simple proximity logic: if user has selected a neighborhood, pick the feed serving that district; otherwise pick the freshest-event's district

### Drill-in modal

A small `(⋯)` icon in the strip opens a modal showing all scanner feeds grouped by service (SFPD/SFFD/EMS) — same content as the old dedicated view, just as a popup. This preserves the discoverability of the full feed list without giving it a route.

### Why launcher-only and not embedded

Per the brainstorm decision documented in `feedback_scanner_as_soundtrack.md` (to be written post-brainstorm): scanner audio is best consumed in a separate tab so the visual data view doesn't compete with the audio for the user's foreground attention. The user-stated mental model: "scanner running in one tab, the map in another" — leans into the browser as a workshop rather than trying to recreate everything in one viewport.

## Freshness Honesty Layer

A two-row chip strip near the top of the chrome (just below the header):

```
DATA REFRESH ·  911 ⚡1m  ·  Fire/EMS 4h  ·  311 5h  ·  Parking 9h  ·  Police 22h
EVENT LAG    ·  911  7h   ·  Fire/EMS 12h  ·  311 15h ·  Parking 17h ·  Police 39h
```

- **Row 1: DATA REFRESH** — how long ago Socrata last published new rows (`rowsUpdatedAt` from views metadata)
- **Row 2: EVENT LAG** — how old the freshest event in each dataset is (current `MAX(dateField)`)

These are *different stories*. Row 1 tells the user "the pipeline is moving"; Row 2 tells them "but it's behind." Both matter.

### Color coding

- Green tone (moss): < 30 min
- Neutral paper: 30 min – 6 h
- Ochre warning: 6 h – 24 h
- Brick concern: > 24 h

### Methodology popover

Click anywhere on the freshness strip → opens a methodology popover explaining:
- What `rowsUpdatedAt` and `MAX(dateField)` mean
- Why they can diverge
- Where to find the underlying Socrata metadata if a journalist wants to verify

This is part of DataDiver's stated commitment to a methodology whitepaper (see `project_methodology_whitepaper.md`).

## Testing Approach

### Data engine tests (unit)

- `useLast48Window` — mock `fetchDataset`, verify polling cadence per dataset, dedup logic on overlapping fetches, 48h eviction, error handling per dataset
- `useAnomalyBaseline` — verify z-score calculation against known baselines, edge cases (zero variance, missing neighborhood, < 12 weeks of history)
- `usePollCadence` — verify visibility pause/resume, jitter distribution, exponential backoff on failure

### Component tests (integration)

- Mode toggle URL state round-trip — clicking HOTSPOTS updates URL; pasting URL into new tab loads HOTSPOTS directly
- Dataset filter behavior — toggling chips updates `useLast48Window` opts and recomputes correctly
- Scrubber drag — verify time cursor updates `visibleEvents` filter correctly
- Kiosk director — scene sequencing fires in order; pause flag stops it; auto-extend on quiet data

### Visual / manual

- Live Vercel preview verification before merge (DataDiver convention)
- Kiosk landscape AND portrait visual check on the user's 55" portrait monitor (per `user_background.md`)

### Socrata schema-drift detection

The `useLast48Window` hook must NOT silently fail if a dataset schema changes. On non-200 responses or unexpected row shapes, the dataset is marked as `error` in the `freshness` map and visually de-emphasized in chrome ("⚠️ 311 unavailable" instead of disappearing). This is the same lesson PR #29 taught us about `date_diff_mm` retirement.

## Out of Scope (explicit)

- **Persisting the buffer across reloads.** Each session starts fresh. Browser refresh = re-poll. Acceptable given Socrata's relaxed rate limits.
- **Push notifications when fresh events arrive.** Could be a later polish layer; out of MVP scope.
- **Custom date-range override.** The Last 48 always uses the last 48h. Users wanting longer windows go to the per-dataset views.
- **Multi-city support.** SF only. Future hypothetical generalization is not designed here.
- **OmniSearch integration on this view.** The existing OmniSearch entity-search is not surfaced from Last48; deferred to the OmniSearch v2 spec.
- **Server-side push (SSE/WebSocket).** Adding a backend is out of scope — DataDiver is client-only against Socrata.
- **Health/uptime dashboard for datasets.** A future polish layer (referenced in `project_methodology_whitepaper.md`).

## Open Questions / Deferred Decisions

1. **Scene mix in KIOSK** — the 6-scene script above is a first pass. Should be exercised on the user's 55" portrait monitor for ~30 min to test whether the rotation feels right or repetitive. Likely to need adjustment after observation.
2. **HOTSPOTS multi-dataset z-score weighting** — the proposed inverse-variance weighting may be unintuitive. Alternative: each dataset gets equal weight, and we display the average z-score. Validate empirically before deciding.
3. **Freshness chip "concern threshold"** — when does a dataset's lag become alarming enough to suppress it from FLOW? Current proposal: never suppress, always show with color-coded chip. But if a dataset goes 48+ h without a refresh, maybe it should drop out of FLOW. Defer until we see real-world drift.
4. **Tier-2 inclusion** — should Police Incidents always be on, off by default, or user-toggleable? Current proposal: off by default; chip available to enable. May need adjustment after user testing.
5. **Mobile / phone form factor** — design above targets desktop and kiosk. Phone-sized viewport (<480px) probably renders as FLOW-only with the scanner strip moved into a settings drawer. Not designed in detail here; can be a follow-up.
