# Elections UI — precinct fill, neighborhood frame, era-aware geometry

**Date:** 2026-07-14
**Status:** SHIPPED — implemented via `docs/superpowers/plans/2026-07-14-elections-ui.md`
(14 tasks) and merged in PR #112 (`752e828`, 2026-07-14 evening). Two user-approved

**Depends on:** the results generator shipped in `60e302c`/`8632cf7` (see

**Companion knowledge:** `docs/data-insights.md` → Elections (six findings)

## What this phase does

Wires the certified precinct + neighborhood results now sitting in
`public/data/elections/results/` into the Elections view. The map becomes a real
precinct choropleth (the 2024 presidential spans 38.9 points precinct to
precinct); neighborhood panels stop showing citywide numbers under a
neighborhood heading; geometry follows each election's era across the 2022
redistricting break.

Approved decisions, in one place:

1. **Precinct fill + neighborhood frame.** The fill is ALWAYS precincts.
   Neighborhoods are boundary lines + click targets on top — selecting one
   highlights it and opens its certified numbers. No grain toggle, no
   zoom-adaptive cross-fade.
2. **Results paint = leader hue × lead strength.** Hue answers *who leads
   here*; intensity answers *how decisively*. Works identically for 2-way and
   16-way races.
3. **Propositions at precinct grain are IN scope** (yes/no is the one honest
   two-way case; the existing `measureColor` diverging ramp is exactly right).
4. **Time Machine × precinct fill is IN scope.** Turnout is the flagship
   cross-election story; era geometry swaps at the 2022 boundary.
5. Architecture is composable modules (the `Last48/modes/` pattern), not
   growing `Elections.tsx` in place (938 lines already) and not a full rewrite.

## Data contracts (verified against the emitted files — do not re-derive)

All files are minified single-line JSON, byte-deterministic, regenerated by
`node scripts/build-election-results.mjs` (reconciliation-gated; see the
pipeline spec). Client code treats them as read-only truth.

```
public/data/elections/results/<dateCode>/
  summary.json                 (pre-existing citywide file — unchanged)
  neighborhoods.json           168 KB for 2024
  precincts/_turnout.json      44 KB for 2024
  precincts/<raceId>.json      ~170 KB for a citywide race; one per race
```

`neighborhoods.json`:

```jsonc
{
  "dateCode": "20241105",
  "scheme": "analysis41",        // or "legacy26" for 20201103 / 20220607
  "neighborhoods": {
    "CASTRO/UPPER MARKET": {
      "registered": 17729, "ballots": 15618, "turnout": 0.8809,
      "races": { "<raceId>": { "votes": { "<CANDIDATE>": 1728, ... }, "total": n } }
    }, ...
  }
}
```

`precincts/_turnout.json`:

```jsonc
{
  "dateCode": "20201103",
  "era": "prec_2012",            // or "prec_2022"
  "precincts": {
    "1101":      { "ids": ["1101"],          "registered": 605, "ballots": 529, "turnout": 0.874 },
    "1104/1105": { "ids": ["1104","1105"],   ... },              // consolidated row
    "7055":      { "ids": [], "unmapped": true, ... }            // no geometry exists
  },
  "suppressed": { "registered": 1215, "ballots": 983 },          // in dsov, absent from sov
  "unmapped":   { "precincts": 12, "registered": 9544, "ballots": 8094 }
}
```

`precincts/<raceId>.json`:

```jsonc
{
  "dateCode": "20241105", "raceId": "president-and-vice-president",
  "title": "PRESIDENT AND VICE PRESIDENT", "era": "prec_2022",
  "precincts": { "<label>": { "votes": { "<CANDIDATE>": n, ... }, "total": n } }
}
```

**The join contract:** race files key on the row *label* (`"1104/1105"`), which
matches `_turnout`'s keys. Only `_turnout` carries `ids` — the geometry join is
`label → _turnout.precincts[label].ids → era geometry feature ids`. Every
member id of a consolidated label paints with that label's values. `unmapped`
rows have `ids: []` and render nowhere, on purpose. `raceId` slugs agree with
`summary.json`'s race ids (generator-enforced), so the existing race selection
state keys both files.

**Neighborhood names are UPPERCASE in dsov-derived data** (`CASTRO/UPPER
MARKET`) while the vendored boundary geojson uses title case (`Castro/Upper
Market`). Normalize once at a module boundary
(`normalizeNhood = s => s.toUpperCase()` on the geometry side, or a shared
`nhoodKey()`), never ad-hoc at call sites.

## Task 1 — geometry vendoring (build-time)

New `scripts/build-precinct-geometry.py`, sibling of
`scripts/build-neighborhood-boundaries.py` (same shapely dissolve / precision-6
rounding / sliver rules — read that script's docstring for the buffering trap:
do NOT weld slivers by morphological close, it doubles the file).

Reads `data/elections-src/prec_2012.geojson` + `prec_2022.geojson` (gitignored
sources; re-fetch with `node scripts/fetch-election-sources.mjs`). Emits to
`public/data/elections/geo/`:

- `prec-2012.geojson` — 605 features, props stripped to `{ id, nhood }`
  (`id` = `prec_2012` field as string, `nhood` = `neighrep`)
- `prec-2022.geojson` — 514 features, props `{ id, nhood }`
  (`id` = `prec_2022` as string, `nhood` = `neigh22`)
- `legacy-neighborhoods.geojson` — dissolve `prec_2012` by `neighrep` →
  features `{ nhood }` (drop sub-0.1% sliver parts, same as the neighborhoods
  script). Don't pin the count in prose — gate it in-script: feature count ==
  distinct `neighrep` values in the source, AND every neighborhood key in
  `results/20201103/neighborhoods.json` matches a feature via `nhoodKey()`.

Expected sizes ~600–900 KB per precinct file (from 2.1–2.4 MB source).
Sanity gates in-script (fail loudly, don't warn): feature counts 605/514,
every emitted `id` unique, every `prec-2022` id present in
`results/20241105/precincts/_turnout.json`'s expanded ids and vice versa
(minus the unmapped allow-list for 2012).

The existing `public/data/elections/geo/precincts.geojson` (1.0 MB, Prec_2025
props) and `precinct_neighborhood_map.json` stay until their last consumer
(`usePrecinctBoundaries` in the sidebar) is retired by Task 4, then delete
both + the hook if nothing else imports it (`grep` first — `useMapCameraPresets`
and `pointInPolygon` consume *neighborhood* boundaries, not these).

## Task 2 — data hooks

Extend the `useStaticJSON` module-cache family in
`src/hooks/useElectionResults.ts` (same file; they're small):

```ts
useElectionGeo(era: 'prec_2012' | 'prec_2022' | null)        // → FeatureCollection
useLegacyNeighborhoodGeo(enabled: boolean)                   // legacy frame
usePrecinctTurnout(dateCode: string | null)                  // → _turnout.json
usePrecinctRace(dateCode: string | null, raceId: string | null) // lazy per race
useNeighborhoodResults(dateCode: string | null)              // → neighborhoods.json
```

All cached module-level (the existing pattern). `usePrecinctRace` refetches
only on race change — browsing races in one election is one ~170 KB fetch per
race, cached thereafter.

## Task 3 — precinct fill (`src/views/Elections/map/`)

New directory, Last48-`modes/` style:

- `precinctPaint.ts` — PURE, Vitest-tested. Exports:
  - `leaderOf(votes: Record<string,number>): { name, share, lead } | null`
    (share = leader/total; lead = leader − runnerUp, both as fractions;
    null when total = 0)
  - `resultsFill(leader, colorMap): { color, opacity }` — leader's palette
    color; opacity stepped by share: <0.34 → 0.25, <0.5 → 0.4, <0.65 → 0.55,
    else 0.7 (four steps, not a continuous ramp — steps read as "levels of
    decisiveness," continuous reads as noise at 500-polygon scale)
  - `propFill(yesShare)` — `measureColor` diverging (brick→paper-300→moss),
    opacity fixed 0.55
  - `turnoutFill(turnout)` — existing `turnoutColor` ramp, opacity 0.55
  - `marginFill(lead)` — indigo 400→600 intensity (the `marginColor` scale)
  - `isProposition(raceId, title)` — props are races whose title matches
    `/^PROPOSITION|^MEASURE/i`; used to pick propFill over resultsFill
- `PrecinctFillLayer.tsx` — builds one FeatureCollection per (election, race,
  mode): expand labels→ids via `_turnout`, attach precomputed `fillColor` /
  `fillOpacity` per feature (500 features is cheap; no feature-state
  machinery). Renders via `useMapLayer` with `{ belowLabels: true }` (house
  rule: dense fills under basemap labels). Outline: 0.5px hairline
  `rgba(255,255,255,0.12)` dark / espresso-tint light (the underlay idiom).
- `NeighborhoodFrameLayer.tsx` — era-correct boundary lines above the fill
  (NOT belowLabels), indigo `ACCENT`, width 1 / opacity 0.35; selected
  neighborhood: width 2 / opacity 0.9 + slight fill lift on its precincts.

Mode → paint routing: `results` uses `propFill` when the active race is a
proposition, else `resultsFill`; `turnout`/`margin` unchanged by race type.

## Task 4 — panels, sidebar, honesty

- **`NeighborhoodElectionPanel` gets real data** from
  `useNeighborhoodResults`: certified registered / ballots / turnout, the
  active race's votes in that neighborhood (top candidates + share), and a
  `PositionScale` placing the neighborhood's turnout against citywide (the
  comparison-framing convention — citywide stays the canvas). Era-correct:
  a 2020 selection shows the legacy name and its legacy-scheme numbers.
- **Sidebar Neighborhoods tab** lists `Object.keys(neighborhoods.json)` for
  the active election (era-correct vocabulary for free), each row showing
  turnout + PositionScale-style delta vs citywide. The old "N precincts"
  sub-label is DROPPED (it was never load-bearing and would need a geometry
  join to keep). This retires `usePrecinctBoundaries` from Elections.
- **Kill the fallback artifacts**: `hasCitywideFallback`, the "Citywide
  results — neighborhood-level data coming soon" pill, and the tooltip
  citywide caveat — replaced by real per-precinct tooltips: neighborhood
  name, leader + share ("Harris — 7 in 10 votes"), turnout ("74% turned
  out"), votes cast. Dejargoned per the pulsePhrase principle: no σ, no
  "ratio", spell out what a number means.
- **Precinct selection**: click precinct → `?precinct=<label>` URL param →
  compact `DetailPanelShell` (top-right, house pattern) with the precinct's
  full candidate breakdown, its parent neighborhood (click-through to select
  it), registered/ballots/turnout. Clicking empty water / pressing X clears.
  Neighborhood click targets remain via the sidebar and the frame layer;
  when both could match, precinct wins (it's the finer target).
- **The honesty chip** (legacy elections only): a small `MapLoadingIndicator`-
  style corner pill — "12 precincts (9,544 voters) can't be drawn for this
  election" → links `/about#elections`. Numbers come from `_turnout.unmapped`,
  not hardcoded. Suppressed residual: one footer line in the precinct panel
  ("SF withholds N tiny precincts for ballot secrecy — 1,215 voters in this
  election are counted citywide but not shown per precinct"), from
  `_turnout.suppressed`.

## Task 5 — Time Machine × precinct fill

- Timeline scrubbing (existing `useElectionTimeline`) drives the fill.
  **Turnout mode is the flagship**: preload all six `_turnout.json` (~270 KB
  total) + both era geometries when Time Machine activates, then scrub with
  zero fetches.
- In Results mode, each election paints its own auto-picked race (existing
  `activeRace` auto-pick), race title shown plainly on the timeline HUD.
  Race files are fetched lazily as the scrub crosses each election;
  a not-yet-loaded election shows the turnout paint for that beat rather
  than a blank (progressive, never empty).
- **Era swap at the 2022 boundary**: fade fill-opacity to 0 over ~150 ms, swap
  source data + geometry, fade back ~150 ms. Calm, civic-observatory register —
  no morphing, no camera moves. The neighborhood frame swaps in the same beat
  (41 ↔ 26). A one-line HUD note on the legacy side: "boundaries as drawn for
  this election era."
- Reduced-motion: fades become instant swaps (existing convention).

## Task 6 — stat cards

Cards become selection-aware, comparison-framed (EmergencyResponse PR #23
pattern): no selection → citywide (turnout, ballots, winner, margin — as
today, from `summary.json`); neighborhood selected → its certified turnout +
ballots with PositionScale vs citywide; precinct selected → its turnout +
leader. Cards never lose the citywide reference tick.

## Out of scope

- RCV round-by-round on the map (no round-level precinct data exists in DSOV).
- Mobile redesign beyond existing MapSidebar/DetailPanelShell behavior.
- Any change to `summary.json` or the generator (data layer is frozen; UI is
  read-only against it).
- Deriving modern-scheme neighborhoods for pre-2022 elections (impossible;
  see the pipeline spec).

## Conventions that bind this work (for a fresh session)

- Dense fills go `belowLabels: true`; heatmap-bright colors are for dark
  basemaps; hairline outlines from the underlay idiom.
- `useMapLayer` try-catch + setTimeout retry is the ONLY reliable Mapbox v3 +
  React pattern; MapView calls `onMapReady` immediately.
- Detail panels: `DetailPanelShell`, top-right, click-driven (never hover-dwell).
- Reader-facing copy is dejargoned (pulsePhrase principle) — no σ, no ratios.
- Comparison framing: citywide is the canvas; selection is context layered on.
- Earth-tone only; candidate colors from `buildCandidateColorMap` (guard test
  pins the palette to tokens.css).
- Verify with full `pnpm build` (devman wrapper), never `tsc -b` alone.
